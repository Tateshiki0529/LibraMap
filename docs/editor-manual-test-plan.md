# LibraMap Editor 手動テスト手順書

## 目的

LibraMap Editor MVPが、書架マップJSONを安全に編集・保存・検証できることを確認する。

対象範囲は、フロア編集、オブジェクト編集、書架グリッド設定、NDC範囲編集、JSON保存、LibraMap本体との連携確認である。

## 前提条件

- PR #2 または `feature/editor-mvp` のコードで実施する
- Python仮想環境を有効化済み
- `pip install -r requirements.txt` を実行済み
- `libramap/data/schema.json` が存在する
- テスト前に作業ツリーがクリーンである

## テスト前準備

1. ブランチと差分を確認する。

```powershell
git status --short --branch
```

2. 自動テストを実行する。

```powershell
python -m unittest discover -s tests
```

3. Editorを起動する。

```powershell
python -m libramap_editor.main
```

## 基本表示テスト

### E-01 Editor起動

1. Editorを起動する。
2. 画面左にフロア一覧が表示されることを確認する。
3. 画面左下にオブジェクト一覧が表示されることを確認する。
4. 中央にプレビューが表示されることを確認する。
5. 右側にオブジェクト編集フォームとNDC範囲表が表示されることを確認する。

期待結果:

- アプリが落ちない
- `1f` / `2f` 相当のフロアが選択できる
- 既存書架や禁帯出エリアがプレビューに表示される
- ボタン、入力欄、表の文字が視認できる

### E-02 フロア切り替え

1. フロア一覧で別フロアを選択する。
2. オブジェクト一覧が選択フロアの内容に切り替わることを確認する。
3. プレビューが選択フロアに切り替わることを確認する。

期待結果:

- 選択したフロアのオブジェクトだけが表示される
- プレビューが破綻しない

## オブジェクト編集テスト

### E-03 既存書架の属性編集

1. 書架オブジェクトを選択する。
2. `X` または `Y` を変更する。
3. `幅` または `高さ` を変更する。
4. プレビューが更新されることを確認する。
5. `検証` を押す。

期待結果:

- 入力値がフォームに保持される
- プレビュー上の位置またはサイズが変わる
- 検証結果が問題なしになる

### E-04 書架以外の属性編集

1. `desk`、`stairs`、`restricted`、`return_box` のいずれかを選択する。
2. 座標またはサイズを変更する。
3. `段数`、`列数`、NDC範囲表が編集対象外になることを確認する。

期待結果:

- 書架以外ではNDC範囲表を編集できない
- プレビューが更新される

### E-05 オブジェクト追加

1. フロアを選択する。
2. `オブジェクト > 追加` を押す。
3. 新しい書架が追加されることを確認する。
4. ID、座標、サイズを変更する。
5. `検証` を押す。

期待結果:

- 新規書架がオブジェクト一覧に追加される
- ID重複がなければ検証が通る
- プレビューに新規書架が表示される

### E-06 オブジェクト削除

1. テスト用に追加したオブジェクトを選択する。
2. `オブジェクト > 削除` を押す。
3. 一覧とプレビューから消えることを確認する。

期待結果:

- 対象オブジェクトだけが削除される
- 他のオブジェクトは残る

## NDC範囲編集テスト

### E-07 NDC範囲追加

1. 書架を選択する。
2. `NDC範囲 > 行追加` を押す。
3. 追加行の `段`、`開始列`、`終了列`、`NDC開始`、`NDC終了` を入力する。
4. `検証` を押す。

期待結果:

- 範囲が表に追加される
- 書架の段数・列数内の値なら検証が通る

### E-08 NDC範囲エラー

1. 書架を選択する。
2. 範囲表で存在しない段番号、または列範囲外の値を入力する。
3. `検証` を押す。

期待結果:

- 検証エラーが表示される
- アプリが落ちない
- エラー修正後に検証が通る

### E-09 NDC範囲削除

1. 書架のNDC範囲行を選択する。
2. `NDC範囲 > 行削除` を押す。
3. 対象行が削除されることを確認する。

期待結果:

- 選択行だけが削除される
- JSON構造が壊れない

## 保存テスト

### E-10 別名保存

1. `別名保存` を押す。
2. `scratch/editor-test-schema.json` などGit管理外または一時ファイルへ保存する。
3. 保存したJSONを再度 `開く` で読み込む。

期待結果:

- JSONがUTF-8で保存される
- 再読み込みしてもフロア、オブジェクト、NDC範囲が保持される
- 保存後に検証が通る

### E-11 本体連携確認

1. Editorで保存したJSONを `libramap/data/schema.json` として配置する。
2. LibraMap本体を起動する。

```powershell
python -m libramap.main
```

3. 既存テストISBNをスキャンする。

期待結果:

- LibraMap本体が起動する
- 配架判定が落ちない
- プレビューとレシート描画が破綻しない

## 異常系テスト

### E-12 ID重複

1. 既存オブジェクトと同じIDを別オブジェクトへ入力する。

期待結果:

- エラーが表示される
- JSONが壊れない
- 元のIDに戻せる

### E-13 不正JSON読み込み

1. 壊れたJSONファイルを用意する。
2. `開く` で読み込む。

期待結果:

- エラーが表示される
- 現在編集中の内容が意図せず破壊されない

## specs.md 20.0.6 差分理解

今回追加された `20.0.6 AIモデル使い分け方針` は、実装をCodexに丸投げせず、作業種別ごとにLLMの役割を分けるための運用ルールである。

Editor開発では以下を守る。

- Codexは `feature/*` ブランチで通常実装、テスト、README、手順書追加を担当する。
- 仕様判断、MVP範囲判断、UI/UX判断は人間または上位レビューで確認する。
- 仕様にないドラッグ配置や高度な自動整列は、MVPでは実装せず次フェーズ候補に留める。
- JSON構造変更時は後方互換性と本体側の読み込みを確認する。
- 実装後は手動テスト手順書と自動テストの両方を更新する。

確認事項:

- `20.0.6` が `20.0.4` より前に挿入されているため、仕様書整理時に番号順の修正候補とする。
- `20.1 MVP方針` 見出しが重複しているため、同じく仕様書整理時に修正候補とする。

## 判定基準

- E-01からE-10まで通ればEditor単体MVPは合格
- E-11まで通ればLibraMap本体連携込みで合格
- JSON保存後に本体が起動不能になる場合は不合格
- GUI上で文字が読めない、誤操作しやすい箇所があれば要改善
