# 図書館返却支援システム 実装指示書 v1

## 1. 概要

### 1.1 システム目的
本システムは、図書館返却業務における司書の作業負荷軽減を目的とした「返却支援システム」である。

返却された資料のISBN-13およびJANコードを利用し、国立国会図書館サーチ（NDL Search API）および各図書館の蔵書データベースを参照することで、日本十進分類法（NDC）に基づく書架位置を特定し、視覚的に返却位置を提示する。

本システムは「完全自動配架」を目的とせず、司書による最終判断を前提とした支援システムとして設計する。

---

## 2. システムコンセプト

### 2.1 基本思想
- 厳密な本の位置管理は行わない
- 「おおよその返却位置」を視覚的に提示する
- 司書による最終確認を前提とする
- 図書館ごとのローカルルールに柔軟対応する
- コード変更ではなくGUI設定で運用変更可能とする

### 2.2 対応対象
対象とする資料：
- 館内蔵書DBに登録されている資料
- 恒久配置されている通常書架資料
- 固定配置の禁帯出資料

対象外とする資料：
- 相互利用資料
- 未登録資料
- 特集棚資料
- 期間限定展示資料
- 配架待ちワゴン資料
- 修理中資料
- 一時移動資料

## 3. システム構成

### 3.0 関連プロジェクト構成
本システムは以下関連プロジェクト群で構成する。

| プロジェクト | 役割 |
|---|---|
| LibraMap | 配架支援本体 |
| LibraMap Editor | 書架マップ編集 |
| LibraMap Builder | 蔵書DB生成・検証・デバッグ支援 |

### 3.0.1 LibraMap Builderについて
LibraMap Builder は、本番運用システムではなく、開発・検証・保守支援を目的とした補助ツール群として位置付ける。

主用途：
- ISBN大量投入
- NDL Search API検証
- テストデータ生成
- キャッシュ確認
- NDC確認
- DB生成補助
- CSVインポート検証
- デバッグ支援

### 3.0.2 責務分離
LibraMap本体は「閲覧・配架支援」に責務を限定し、DB生成・加工・検証はBuilder側へ分離する。

これにより以下を実現する。

- 本番システムの軽量化
- 誤スキャンによるDB汚染防止
- テストデータ分離
- API検証分離
- 実験的処理の隔離
- 保守性向上

### 3.0.3 Builder運用方針
Builderは主に開発者・管理者向けツールとして利用する。

通常司書運用では直接利用しない想定とする。

### 3.0.4 Builderデータ管理
Builderで生成したデータは、レビュー・確認後に本番用ローカル蔵書DBへ反映する。

Builder側で直接本番DBを書き換えない構成を推奨する。

## 7. NDL Search API仕様

### 7.1 利用API
NDL Search OpenSearch APIを利用する。

### 7.2 検索方式
ISBN検索を使用する。

### 7.3 取得対象項目
| 項目 | 用途 |
|---|---|
| title | タイトル表示 |
| creator | 著者表示 |
| publisher | 出版社表示 |
| isbn | 一致確認 |

### 7.4 NDC取得仕様
- NDC(10)を優先
- 存在しない場合はNDC(9)

### 7.5 API失敗時
以下順序でフォールバックする。

1. ローカル蔵書DB
2. SQLiteキャッシュ
3. NDL Search API
4. 手動対応

## 8. ローカル蔵書DB仕様

### 8.1 目的
ローカル蔵書DBは本システムの中核データベースとして利用する。

NDL Search APIはローカル蔵書DB構築および補助情報取得用途として利用し、最終的な配架・運用判定はローカル蔵書DBを基準とする。

ローカル蔵書DBは以下用途で利用する。

- 館内蔵書確認
- 相互利用資料除外
- 禁帯出判定
- ローカル運用ルール対応
- 書架位置情報保持
- 利用者向け検索システム連携
- 書架ナビゲーション

### 8.1.1 基本思想
本システムでは、司書による手動登録作業を極力削減することを目的とする。

そのため、ISBN入力時にNDL Search APIから書誌情報を取得し、ローカル蔵書DBへ補助入力可能な構成とする。

図書館側は不足情報のみ補完する運用を想定する。

### 8.1.2 データ同期方針
ローカル蔵書DBは以下情報を保持する。

| 項目 | 取得元 |
|---|---|
| タイトル | NDL Search API |
| 著者 | NDL Search API |
| 出版社 | NDL Search API |
| ISBN | NDL Search API |
| NDC | NDL Search API |
| 書架位置 | 図書館ローカル設定 |
| 禁帯出 | 図書館ローカル設定 |
| 運用フラグ | 図書館ローカル設定 |

### 8.1.3 利用者検索システム対応
本システムは将来的に利用者向け蔵書検索システムへの転用を想定する。

利用者検索時は以下機能を提供可能とする。

- 書籍検索
- 書架位置表示
- フロアマップ表示
- 書架ロケーションナビ
- 禁帯出表示
- 配置エリア可視化

### 8.2 必須仕様
館内蔵書DBに存在しない資料は、自動配架支援対象外とする。

### 8.3 対象外資料印字
対象外資料の場合、以下内容を印字する。

- 対象外資料
- 要手動返却確認
- 相互利用・未登録資料の可能性

## 9. 配架判定仕様

### 9.1 基本方式
NDC範囲一致による書架判定を行う。

### 9.2 判定思想
厳密な本位置は管理せず、司書が視覚的に判断可能な範囲案内を行う。

### 9.3 優先順位
以下優先順位で判定する。

1. 禁帯出ルール
2. 個別例外
3. 詳細NDC
4. 通常NDC

### 9.4 最長一致方式
最も詳細なNDC範囲を優先採用する。

例：
- 913 より 913.6 を優先
- 913.6 より 913.63 を優先

### 9.5 NDC管理方式
NDCは数値ではなく文字列として管理する。

理由：
- float比較誤差防止
- 桁保持
- 詳細分類対応

## 20. 開発方針

### 20.0 開発体制
本プロジェクトは、Antigravityを利用したLLM-based開発を前提として進行する。

開発における役割分担は以下とする。

| 役割 | 内容 |
|---|---|
| LLM | 実装支援・コード生成・補助設計 |
| プロジェクトマネージャー | デバッグ・仕様判断・UI/UX判断・グラフィック判断 |

### 20.0.1 開発原則
- 生成コードを無条件に採用しない
- 必ず人間によるレビューを実施する
- GUI・視認性・運用性は人間判断を優先する
- 実運用を前提とした可読性を重視する
- 暗黙実装を避け、仕様ベース開発を徹底する

### 20.0.2 Git/GitHub運用方針
本プロジェクトではGitHubを利用したGit管理を必須とする。

以下を開発ルールとして定義する。

#### 必須事項
- 全変更をGitで管理する
- 直接masterブランチへpushしない
- 機能単位でブランチを分離する
- Pull Requestベースで統合する
- 変更理由をコミットメッセージへ明記する
- 生成コードも必ずレビュー対象とする

#### 推奨ブランチ構成
| ブランチ | 用途 |
|---|---|
| master | 安定版 |
| develop | 開発統合 |
| feature/* | 機能開発 |
| fix/* | バグ修正 |
| experimental/* | 実験実装 |

#### コミットメッセージ推奨形式
```text
feat: 書架マップ描画機能追加
fix: NDC比較処理修正
refactor: GUI構造整理
```

#### GitHub運用推奨事項
- Issuesによるタスク管理
- Pull Requestレビュー必須
- スクリーンショット添付推奨
- GUI変更時は画像比較を実施
- リリースタグ管理を行う

### 20.0.3 LLM利用時の注意事項
LLMによるコード生成時は以下を徹底する。

- 生成コードをそのまま本番投入しない
- ライブラリ依存を明示確認する
- セキュリティ上危険なコードをレビューする
- ESC/POS制御は実機確認を行う
- GUI生成コードは視認性確認を行う
- JSON/YAML構造変更時は後方互換性を確認する

### 20.0.4 デバッグ方針
- 実機バーコードスキャナで検証する
- 実レシートプリンタで検証する
- API失敗時を必ずテストする
- キャッシュ動作を確認する
- 書架マップの誤判定を重点確認する
- GUI誤操作耐性を確認する

### 20.0.5 UI/UX方針
本システムは公共施設利用を想定するため、以下を重視する。

- 高視認性
- 高齢利用者への配慮
- 直感的操作
- 誤操作防止
- 大型文字表示
- 強調表示
- 低学習コスト

## 21. システムの責務

本システムは司書を置き換えるものではなく、返却業務を支援する補助システムである。

最終的な配架判断は司書が行うものとする。
